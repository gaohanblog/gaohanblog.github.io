09-单例模式

发表于 2019-09-07 更新于 2020-06-24
本文字数： 6k 阅读时长 ≈ 5 分钟

单例模式看似简单，从他的定义是确保一个类只有一个实例，并且自行实例化并向整个系统提供整个实例。但是遇到多线程，序列化时反射，就会很难麻烦，本文我们来探讨下这些问题。

定义

  • 确保类只有一个实例，并提供全局的访问。
  • 优点 只有一个实例，减少了内存的开销。全局的访问点优化和共享资源的访问。
  • 缺点 没有接口，扩展很困难

饿汉式

这种写法比较简单，在类加载的过程中直接完成实例化，避免了延迟加载。但是一定程度上，假如没有用到这个实例，就会造成内存的浪费。对于的实例化过程，也可以放在静态代码块中。效果和优缺点基本和上面一样。

懒汉式

懒汉式和饿汉式的区别在于，懒汉式在用到这个类的时候才去创建这个类的实例，避免了内存浪费，但是懒汉式在多线程下当多个线程进入（if-else）代码块，便会产生多个实例。所以尽量在单线程下使用该方法。

双重锁检查–禁止指令重排序

双重锁检查方法很好的避免了多线程的安全问题，并且实现了延迟加载，内存浪费的问题。

我们先看下volatile关键字，来防止指令重排序，因为instance = new LazyDoubleCheckSingleton();的执行过程：

  • 第一步，分配内存给对象
  • 第二步，初始化对象
  • 第三步，设置instance指针指向对象内存空间的地址。
    但是第二步和第三步可能会被颠倒顺序。

比如当一个线程t1进来执行new，即刚进行第一步的时候，这时候instance还没有指向对象的内存地址，所以instance指向为null。

假如第一个时间片结束，第二个线程t2进来了，首先判断instance是否为null，因为在不加volatile关键字情况下第二步和第三步可能颠倒，所以instance的指向可能不为null，但是对象还没初始化好，这个时候针对这个线程t2可能要报异常了。

此时t2不影响t1的继续进行，再去初始化一个对象，返回初始化好的instance。

但是加上volatile关键字之后，每个线程都可以看到这个共享内存的最新状态了，保证了内存可见性，禁止指令重排序，防止上面出现的指令排序问题。

用volatile关键字修饰的共享变量，在进行写操作的时候，将当前缓存行的数据写回到系统内存中（共享内存）中，这个写回内存的操作会使其他线程缓存这个变量值失效，所以他们需要再去系统内存中同步数据。这样即兼顾了性能达到懒加载的效果，又避免了线程安全问题。

静态内部类

关于静态内部类的原理，我们先看一下一个类的创建过程，当一个类在初始化阶段，jvm都会获取一把锁，这个锁可以同步多个线程对于一个类的初始化，所以基于这个特性，我们给类的内部再加一个类，基于jvm类加载的特性，也可以实现我们上面的延迟加载，线程安全。
